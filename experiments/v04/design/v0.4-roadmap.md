# v0.4 Roadmap

## Goal

`v0.4` should make the current contract-first engine more productive and more
trustworthy before it tries to become broader.

The near-term goal is not "support every external template workflow."
The near-term goal is:

- automate more of the current report-production flow
- make mixed image/text placement trustworthy in real use
- create enough first-party templates to shape the product's visual baseline

## Product lens

The product still centers on:

- inspecting templates that already exist
- exporting a fillable contract
- validating payloads against that contract
- generating deterministic editable output

That means `v0.4` should prioritize work that improves real production
throughput or makes generated PPTX behavior easier to trust.

## Priority order

### 1. Workflow automation

Status: `do now`

Why first:

- it has the biggest immediate effect on productivity
- it creates the operating frame for the rest of `v0.4`
- it can reuse the current runtime without waiting for a larger architecture
  change

Target outcome:

- stable `inspect -> scaffold/fill -> validate -> generate -> review` flow
- branch-local sandbox runs that emit both local artifacts and `.codex`
  evidence
- explicit human review gate before any handoff

Exit criteria:

- a sandbox run can start from a brief and finish with a reviewable `.pptx`
- the run leaves behind a plan, contract, draft payload, artifact, review notes,
  and collector-compatible status evidence
- the flow is clear enough that another tool or teammate can run it without
  guessing the steps

### 2. Mixed-layout hardening

Status: `do next, very high priority`

Why this is above external-template QA:

- real usage already exposed the pain point
- images can be uploaded today, but their actual placement and pairing are not
  yet trustworthy enough
- if mixed `text_image` behavior is shaky, both workflow automation and
  first-party template work lose credibility

Target outcome:

- `text_image` profiling becomes stable for the current editorial path and for
  representative user templates
- image placeholders do not silently become text targets
- caption pairing and placement are predictable enough to demo confidently

Exit criteria:

- representative mixed-layout fixtures inspect into stable contracts
- generation places text, image, and caption in visibly correct destinations
- the user can look at a generated deck and understand where images actually
  land

### 3. First-party template lane

Status: `do after mixed-layout trust improves`

Why this matters now:

- there are no external template suppliers or live consumers yet
- we still need a family of internal templates to shape visual direction and to
  give the model better material to work against
- first-party templates are the fastest way to create usable product examples

Target outcome:

- a small, intentional set of Autoreport-owned templates
- text-first templates first, then richer layouts once mixed-layout behavior is
  trustworthy
- `PptxGenJS` used only as an internal authoring tool

Exit criteria:

- at least one strong text-first template is usable end-to-end
- the template family establishes a consistent visual direction
- authored templates can be handed to the Python runtime without special-case
  runtime behavior

### 4. External template QA

Status: `later`

Why later:

- it is important to the long-term product, but there is no immediate consumer
  pressure yet
- first-party templates and mixed-layout hardening will teach us what should be
  checked before we open this lane wider

What this lane should eventually cover:

- placeholder quality checks
- slot structure diagnostics
- font-risk and theme-risk warnings
- compatibility scoring before generation

### 5. AI-assisted authoring

Status: `defer until the earlier lanes settle`

Current view:

- it may become useful as a payload-drafting helper
- it is not yet clear enough to deserve priority over workflow and layout trust
- it must remain outside the deterministic generation core

### 6. Review and delivery flow

Status: `defer, keep minimal only`

Current view:

- keep explicit review notes and artifact visibility
- do not spend heavily on publish or delivery automation yet
- revisit this once workflow automation is used often enough to expose real
  bottlenecks

## Recommended execution order

1. Finish the current workflow automation lane so it is operationally useful.
2. Harden mixed `text_image` behavior until image placement is trustworthy.
3. Expand the first-party template lane with a small internal template pack.
4. Use what we learn there to define external template QA.
5. Revisit AI-assisted authoring and richer delivery flow only after the first
   four lanes produce real operating evidence.

## What not to do yet

- do not move the runtime to Node.js
- do not make template creation the primary public product story
- do not prioritize external-template QA ahead of mixed-layout trust
- do not build a heavy publish pipeline before the core workflow is used enough
  to justify it

## Current working thesis

The best `v0.4` story is:

`Autoreport becomes a safer, faster contract-first production workflow before it becomes a broader template ecosystem.`
