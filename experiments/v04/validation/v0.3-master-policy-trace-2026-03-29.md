This note records which recent `codex/v0.3-master` orchestration policy changes
were traced against `codex/v0.4-incubator2`, and which of those changes should
actually shape the `v0.4` incubator lane.

## What changed upstream

### 1. Discovery moved to the git worktree registry

The main repo no longer treats a fixed folder list as authoritative. Active
`v0.3` workstreams are discovered from `git worktree list --porcelain`, with
optional `.codex/workstream.json` overrides for `key`, `test_modules`, and
`orchestration_enabled`.

### 2. Policy sync is now a strict master-thread sequence

Tracked policy changes are considered complete only after:

1. the change lands on `codex/v0.3-master`
2. `codex/v0.3-master` is pushed
3. `sync_policy_worktrees.py` rebases active task branches onto the pushed base
4. narrow verification reruns
5. already-remote rebased branches push with `--force-with-lease`

### 3. Retired sibling cleanup is tracked work, not user housekeeping

Old `autoreport_v0.3-*` sibling directories should be cleaned by the tracked
cleanup script. Empty locked directories should first trigger a Codex desktop
restart request before stronger manual deletion is considered.

### 4. `master-next` is now the authoritative branch-specific instruction channel

Once `.codex/master-next.txt` exists for each active worktree:

- branch-specific tasks stay in those local files
- user-facing chat should switch to one shared broadcast
- later master-thread turns should report deltas and blockers instead of
  regenerating branch-by-branch chat handoffs

## What `incubator2` adopts

`incubator2` should keep following only the report-channel-compatible subset of
the upstream policy.

Adopt now:

- emit `.codex/workstream.json`
- emit collector-compatible `.codex/worker-status.json`
- emit `.codex/worker-final.json` only through an explicit finalize step
- document that `.codex/master-next.txt` is separate from workflow automation
  evidence

Do not adopt here:

- automatic inclusion in the current `codex/v0.3-*` discovery flow
- policy sync, rebase, push, or force-push behavior inside the automation plan
- retired sibling cleanup inside the automation plan
- sandbox-generated replacements for the master-thread shared-broadcast model

## Current verdict

No new runtime or sandbox code changes are required beyond keeping the adapter
docs honest.

Reason:

- the collector schema used by `worker-status.json` and `worker-final.json`
  remains compatible with the current `v0.3-master`
- the newer upstream changes are primarily about master-thread orchestration
  behavior, not about the worker report schema itself
- `incubator2` should remain evidence-compatible without pretending to be an
  automatically discovered `v0.3` workstream
