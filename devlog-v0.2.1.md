# Autoreport v0.2.1 개발 일지

기준 날짜: `2026-03-23`

이번 `v0.2.1` 작업은 `v0.1.0`에서 만든 CLI 기반 weekly report 생성기를 바탕으로, 비개발자도 직접 체험할 수 있는 공개 데모 입구를 만드는 데 집중했습니다.  
핵심 방향은 두 가지였습니다. 하나는 YAML 입력과 PPTX 생성을 웹에서도 같은 코어 파이프라인으로 재사용하는 것이고, 다른 하나는 `v0.1` 리뷰에서 드러난 템플릿 호환성 문제를 더 명확하게 정리하는 것이었습니다.

## 이번 버전에서 한 일

- YAML 텍스트를 직접 파싱하는 경로를 추가했습니다.
- 파일 경로 없이도 생성 파이프라인을 재사용할 수 있도록 매핑 기반 생성 함수를 추가했습니다.
- FastAPI 기반 공개 데모 화면을 추가했습니다.
- 브라우저에서 YAML을 붙여 넣고 즉시 `.pptx`를 내려받는 흐름을 구현했습니다.
- weekly report 템플릿 호환성 검사를 추가했습니다.
- 템플릿 형식 오류와 템플릿 읽기 오류를 구분해 사용자 메시지를 더 명확하게 만들었습니다.

## 왜 웹 데모를 먼저 만들었는가

현재 `Autoreport`의 엔진은 이미 동작하지만, 사용 방식은 여전히 CLI와 YAML 파일 중심입니다.  
이 구조는 개발자에게는 충분히 효율적이지만, 비개발자에게는 진입장벽이 높습니다.

그래서 이번에는 기능 자체를 크게 넓히기보다, 기존 엔진을 유지한 채 더 쉬운 입구를 만드는 방향을 택했습니다.  
웹 데모는 새로운 생성 로직을 따로 만드는 것이 아니라, 기존 검증과 PPTX 생성 흐름을 그대로 재사용하도록 설계했습니다.

## 코어 쪽에서 같이 정리한 부분

웹 데모를 추가하면서, 템플릿 관련 안정성도 함께 보강했습니다.

- `--template` 옵션은 임의의 PowerPoint 파일을 모두 지원하는 것이 아니라, 현재 weekly report 레이아웃과 호환되는 템플릿을 전제로 합니다.
- 호환되지 않는 템플릿은 내부 오류로 끝나지 않도록 별도 예외로 분리했습니다.
- 손상된 `.pptx`와 읽을 수 없는 템플릿 파일도 다른 오류로 구분해 처리합니다.

이 작업은 웹 데모를 위한 준비이기도 하지만, CLI 사용성 측면에서도 필요한 정리였습니다.

## App.tsx를 활용해 공개 데모 홈페이지 디자인을 다듬은 방식

이번 작업에서 특히 유용했던 점은, Figma에서 만든 시안을 단순한 이미지로만 보지 않고 `App.tsx` 형태의 구체적인 화면 초안으로 받아볼 수 있었다는 점입니다.

이 `App.tsx` 초안은 공개 데모 홈페이지를 다듬는 데 실제로 큰 기준점이 됐습니다.

- 히어로 문구를 어떻게 짧고 명확하게 보여줄지
- YAML 입력창과 상태 패널을 어떤 비율로 배치할지
- `예제 불러오기`, `PPTX 생성` 버튼을 어떤 우선순위로 둘지
- 따뜻한 배경과 그린 포인트 컬러를 어떤 톤으로 유지할지
- 첫 화면을 과한 대시보드가 아니라 차분한 도구형 화면으로 만들지

이런 요소들을 말로만 조정할 때보다 훨씬 빠르게 합의할 수 있었습니다.

특히 `App.tsx`는 단순한 디자인 참고를 넘어, 실제 구현에서 어떤 정보 구조가 필요한지를 바로 보여주는 역할을 했습니다.  
덕분에 현재 FastAPI 단일 페이지 구현에서도 히어로 문구, YAML 입력 카드, 상태 메시지 패널, 생성 버튼, 하단 안내 문구까지 더 정돈된 형태로 옮길 수 있었습니다.

즉 이번 `v0.2.1`에서는 `App.tsx`를 활용해 공개 데모 홈페이지 디자인을 유의미하게 개선했고, 그 결과를 실제 동작하는 웹 화면에 빠르게 반영할 수 있었습니다.

## Figma를 활용한 협업 방식

이번 작업에서는 화면을 바로 코드로 고정하기보다, 먼저 Figma 시안으로 방향을 잡고 그 결과를 구현에 반영하는 흐름도 함께 시도했습니다.

- Figma에서 공개 데모 화면의 레이아웃과 톤을 먼저 정리
- 그 결과물을 `App.tsx` 형태의 화면 초안으로 구체화
- AI와 함께 문구와 정보 구조를 다듬고, 실제 스키마와 맞지 않는 예시 입력은 수정
- 최종적으로는 현재 FastAPI 단일 페이지 구현에 그 시안의 분위기와 구조를 옮기는 방식으로 반영

이 방식이 유효했던 이유는, 설명만으로 화면을 논의하는 것보다 시안을 기준으로 이야기할 때 훨씬 빠르게 정렬할 수 있었기 때문입니다.  
또 Figma 산출물을 GitHub와 함께 관리하면, 이후 문서화나 추가 구현에서도 어떤 화면을 기준으로 개발했는지를 더 분명하게 남길 수 있습니다.

즉 이번 `v0.2.1`에서는 단순히 웹 데모를 추가한 것뿐 아니라, `Figma -> App.tsx 초안 -> AI 보정 -> 코드 반영`이라는 협업 흐름 자체도 함께 실험한 셈입니다.


즉 `v0.2.1`은 아직 최종 릴리즈는 아니지만, 공개 데모와 코어 안정화를 함께 확인할 수 있는 첫 번째 정리된 개발 기준점이라고 볼 수 있습니다.

## 다음 단계

다음으로는 아래 작업이 자연스럽게 이어질 수 있습니다.
