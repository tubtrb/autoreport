# v0.3 Template Workstreams

This note defines the horizontal work split for the `v0.3` template-aware
engine expansion.
It is contributor-facing planning guidance, not public product copy.

## Why this file exists

The target runtime flow is:

1. user selects a PowerPoint template
2. `autoreport` inspects the template and shows the required input contract
3. a human or another AI fills that contract in YAML or JSON
4. `autoreport` generates a `.pptx` that follows the selected template

That flow is larger than the current fixed weekly-report contract, so work is
split into parallel threads with clear ownership.

## Branch strategy

- Integration base branch: `codex/v0.3-template-engine`
- Each thread branches from that base and merges back into that base
- Keep tasks horizontal: each thread should own one cross-cutting concern
- Do not split by "top half of file" / "bottom half of file"

## Shared guardrails

- Treat repository code and tests as the source of truth
- Preserve the current CLI and web error contracts unless a thread explicitly owns them
- Do not redesign the whole pipeline in one thread
- Prefer adding new generalized paths beside the current weekly path, then migrate deliberately
- If a thread needs a new shared type, document the interface in its PR before asking other threads to depend on it

## Shared target contracts

The threads below should align around these shapes unless the integration owner changes them:

```yaml
template_contract:
  template_id: string
  source_template: string
  slides:
    - role: title | contents | body
      layout_name: string
      title_slot:
        slot_id: string
      text_slots:
        - slot_id: string
          required: true
          orientation: horizontal | vertical | stack
          order: integer
      image_slots:
        - slot_id: string
          required: false
          orientation: horizontal | vertical | stack
          order: integer
```

```yaml
report_payload:
  template_id: string
  title_slide:
    title: string
    subtitle:
      - string
  slides:
    - title: string
      include_in_contents: true
      text:
        - string
      images:
        - path: string
          fit: contain | cover
          caption: string
```

## Contract detail

These details are intentionally explicit so each branch can implement against
the same draft contract without guessing hidden semantics.

### `template_contract`

- `template_id`
  - stable identifier generated from the inspected template
  - should be usable by CLI, web, and cached contract flows
- `source_template`
  - user-facing source path or logical template label
  - should help users confirm which template was inspected
- `slides`
  - ordered list of profiled slide roles that the engine can generate against
  - each entry describes one reusable generation pattern, not every existing slide instance in the source file
- `role`
  - `title`: opening slide pattern
  - `contents`: table-of-contents slide pattern
  - `body`: normal content slide pattern
- `layout_name`
  - exact PowerPoint layout name when available
  - if layout names are duplicated, the exporter should still preserve a deterministic order
- `title_slot`
  - single primary title target for the slide pattern
  - later drafts may allow `null` only if a slide truly has no title behavior
- `text_slots`
  - ordered targets for textual content
  - should capture slot direction strongly enough that another AI can prepare content without seeing the `.pptx`
- `image_slots`
  - ordered targets for images
  - may be empty when the pattern is text-only

### Slot fields

- `slot_id`
  - stable identifier within one template contract
  - should not depend on localized placeholder names alone
- `required`
  - whether the generated payload must provide content for the slot in the happy path
- `orientation`
  - `horizontal`: sibling slots read left to right
  - `vertical`: sibling slots read top to bottom
  - `stack`: same-column stacked slots with stronger top-to-bottom semantics
- `order`
  - deterministic sibling order used by both planning and UI display

### `report_payload`

- `template_id`
  - must match the inspected template contract
- `title_slide`
  - contract for the opening slide
- `title`
  - user-facing deck title
- `subtitle`
  - ordered list so templates with one, two, or more subtitle lines can reuse the same payload shape
- `slides`
  - ordered generated slides after the title slide
- `slides[*].title`
  - canonical section title used by the slide itself and by `Contents`
- `slides[*].include_in_contents`
  - default should be `true`
  - if `false`, the slide is generated but omitted from the table of contents
- `slides[*].text`
  - ordered text blocks to distribute into profiled text slots
- `slides[*].images`
  - ordered image blocks to distribute into profiled image slots
- `slides[*].images[*].path`
  - local file path or future approved source reference
- `slides[*].images[*].fit`
  - `contain` keeps the whole image visible
  - `cover` allows cropping to fill the slot
- `slides[*].images[*].caption`
  - optional text associated with the image when the template exposes a caption slot

## Contract ownership

- `template-contract-export` owns the extracted `template_contract` shape
- `generic-payload-schema` owns the validated `report_payload` shape
- `text-layout-engine` owns how ordered `text` items map into `text_slots`
- `image-layout-engine` owns how ordered `images` map into `image_slots`
- `template-selection-flow` owns how these contracts are shown, downloaded, and accepted in CLI and web flows

## Workstreams

### 1. Template Contract Export

- Branch: `codex/v0.3-template-contract-export`
- Owns:
  - template inspection model
  - PPTX layout and slot extraction
  - JSON and YAML contract export helpers
- Main files:
  - `autoreport/templates/`
  - `autoreport/engine/generator.py` only if a read-only integration hook is needed
  - new tests under `tests/`
- Must not own:
  - CLI UX
  - web UX
  - final image insertion
- Done when:
  - a supplied `.pptx` can be profiled into a structured template contract
  - title, contents, text, and image-capable slots are surfaced with layout name and ordering metadata
  - at least two template fixtures are covered in tests
- Verification:
  - `.\venv\Scripts\python.exe -m unittest tests.test_generator tests.test_pptx_writer`

### 2. Generic Payload Schema

- Branch: `codex/v0.3-generic-payload-schema`
- Owns:
  - generalized YAML and JSON payload schema
  - validation rules
  - typed request models for template-driven content
- Main files:
  - `autoreport/models.py`
  - `autoreport/validator.py`
  - example payloads under `examples/`
  - tests for schema and validation
- Must not own:
  - PPTX slot scoring
  - writer internals
  - UI flow
- Done when:
  - another AI can fill a published YAML or JSON contract without editing code
  - validation errors clearly identify missing, invalid, or extra fields
  - current weekly flow remains valid or has an explicit migration path
- Verification:
  - `.\venv\Scripts\python.exe -m unittest tests.test_validator tests.test_loader`

### 3. Text Layout Engine

- Branch: `codex/v0.3-text-layout-engine`
- Owns:
  - multi-slot text planning
  - horizontal and vertical text-slot ordering
  - continuation and spill rules for text
  - contents slide synchronization with actual slide titles
- Main files:
  - `autoreport/templates/autofill.py`
  - `autoreport/templates/weekly_report.py`
  - `autoreport/outputs/pptx_writer.py`
  - generation and writer tests
- Must not own:
  - schema design
  - image loading and crop policy
  - CLI/web contract display
- Done when:
  - one-column, two-column, and vertically stacked text slots all work
  - `Contents` is generated from actual slide titles and honors `include_in_contents`
  - text can be distributed across `n` slots before continuation slides are created
- Verification:
  - `.\venv\Scripts\python.exe -m unittest tests.test_autofill tests.test_generator tests.test_pptx_writer`

### 4. Image Layout Engine

- Branch: `codex/v0.3-image-layout-engine`
- Owns:
  - image slot profiling
  - picture and object placeholder writes
  - contain and cover fit policy
  - vertical and horizontal image ordering
  - optional caption slot binding
- Main files:
  - `autoreport/templates/autofill.py`
  - `autoreport/templates/weekly_report.py`
  - `autoreport/outputs/pptx_writer.py`
  - image fixtures and tests
- Must not own:
  - schema wording
  - CLI/web interaction flow
  - text-only continuation policy except where image coexistence requires it
- Done when:
  - landscape and portrait images can both be placed into profiled template slots
  - slot order and fit policy are deterministic
  - captions can be attached when the template exposes a caption slot
- Verification:
  - `.\venv\Scripts\python.exe -m unittest tests.test_generator tests.test_pptx_writer`

### 5. Template Selection Flow

- Branch: `codex/v0.3-cli-web-template-flow`
- Owns:
  - CLI template inspection command or equivalent entrypoint
  - web demo flow for template upload or selection
  - contract display to the user
  - skeleton YAML and JSON output for other AI tools to fill
- Main files:
  - `autoreport/cli.py`
  - `autoreport/web/app.py`
  - `README.md` only if user-facing usage changes are finalized
  - docs that explain the new flow
- Must not own:
  - low-level slot heuristics
  - image geometry internals
  - validator internals beyond wiring
- Done when:
  - a user can inspect a template and receive the required input shape before generating
  - CLI and web both expose the same contract structure
  - example template-driven flows are documented
- Verification:
  - `.\venv\Scripts\python.exe -m unittest tests.test_cli tests.test_web_app`

## Merge order

1. `codex/v0.3-template-contract-export`
2. `codex/v0.3-generic-payload-schema`
3. `codex/v0.3-text-layout-engine`
4. `codex/v0.3-image-layout-engine`
5. `codex/v0.3-cli-web-template-flow`

`text-layout` and `image-layout` can run in parallel once the contract-export
and payload-schema interfaces are stable enough.

## Thread handoff checklist

Each thread should close only after it can answer "yes" to all of these:

- Is the owned interface documented in code or tests?
- Are the narrow verification commands passing?
- Are unrelated files left alone?
- Is the integration base branch still the right merge target?
- Did the thread avoid changing public wording unless it owned the flow?
